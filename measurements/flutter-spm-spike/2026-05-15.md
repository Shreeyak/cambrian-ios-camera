# Flutter × Swift Package Manager spike — 2026-05-15

> **Recommended migration path: SPM.** This spike verified the full
> three-target CameraKit shape (Swift / Swift-Cxx-interop / C++20) plus
> `apple/swift-atomics`, Metal shader resources, iOS 26 deployment floor,
> static + dynamic linkage, and multi-plugin dual-mode coexistence — all
> PASS, all substantially simpler than the CocoaPods path. The companion
> doc at `measurements/cocoapods-cxx-spike/2026-05-15.md` remains valid as
> a documented fallback, but there is no remaining technical reason to
> prefer it for the CameraKit Flutter migration.

**Companion to** `measurements/cocoapods-cxx-spike/2026-05-15.md`. The CocoaPods
spike proved the migration path works in CocoaPods; this spike asks whether
Flutter's Swift Package Manager integration is a strictly simpler alternative
for the same shape (Swift / Swift-Cxx-interop / C++20 + `swift-atomics`).

**Question.** Can a Flutter iOS plugin built around SPM (`flutter config
--enable-swift-package-manager`) express CameraKit's three-target shape
verbatim, consume `apple/swift-atomics` as a normal `.package(url:)`
dependency, and ship Metal shader resources via SPM's `process` mechanism —
without the four-pod / vendored-atomics / `Bundle(for:)` overhead the
CocoaPods path requires?

**Verdict.** **PASS, strictly simpler than the CocoaPods path.** The plugin's
`ios/<plugin>/Package.swift` accepts the same three-target structure used in
`eva-swift-stitch/CameraKit/Package.swift` verbatim. `apple/swift-atomics`
is a direct `.package(url:)` dependency — no vendoring. Metal shaders use
`resources: [.process("Shaders")]` + `Bundle.module` — no `Bundle(for:)`
migration. Flutter's auto-generated `FlutterGeneratedPluginSwiftPackage`
umbrella picks up the plugin's iOS-26 platform via a `flutter build`
migration step. No `Podfile`, no `Pods/`, no `:path =>` entries — until
a CocoaPods-only plugin gets pulled in, at which point Flutter cleanly
splits into dual-mode (see Caveat 3 addendum).

All three caveats originally flagged as open have been closed:

- **Caveat 1 (swift build standalone):** Not a meaningful test — the existing
  eva-swift-stitch CameraKit already can't be `swift build`'d (CLAUDE.md §6
  forbids it; SPM defaults to host triple, CameraKit is iOS-only). The
  migration target inherits this constraint; no change in behavior.
- **Caveat 2 (dynamic linkage):** Verified. Setting the plugin's product
  to `.library(name: …, type: .dynamic, …)` produces
  `Runner.app/Frameworks/camera-spike-spm.framework` and the build still
  succeeds. Flutter's static umbrella tolerates depending on a dynamic
  plugin product.
- **Caveat 3 (multi-plugin dual-mode):** Verified. Adding a CocoaPods-only
  sibling plugin (no `Package.swift`) drives Flutter into dual-mode:
  `Podfile`/`Pods/` and `FlutterGeneratedPluginSwiftPackage/` both active;
  `GeneratedPluginRegistrant.m` registers SPM and Pods plugins uniformly.

**Time-box.** ~80 min total: 50 min for the core SPM spike + ~30 min for
the three-caveat addendum.

## What `flutter create -t plugin` produces with SPM enabled

After `flutter config --enable-swift-package-manager` + `flutter create -t
plugin --platforms=ios`, the relevant files are:

```
camera_spike_spm/
├── ios/
│   ├── camera_spike_spm.podspec              # dual-mode default (CocoaPods fallback)
│   └── camera_spike_spm/
│       ├── Package.swift                      # iOS 13 default, single target
│       └── Sources/camera_spike_spm/
│           ├── CameraSpikeSpmPlugin.swift     # FlutterPlugin registrar
│           └── PrivacyInfo.xcprivacy
└── example/ios/
    ├── Flutter/
    │   ├── Debug.xcconfig
    │   ├── Release.xcconfig
    │   └── ephemeral/Packages/                # regenerated on `flutter pub get` + `flutter build`
    │       ├── .packages/FlutterFramework/Package.swift           # Flutter itself is a Swift Package
    │       └── FlutterGeneratedPluginSwiftPackage/Package.swift   # umbrella aggregating plugin products
    └── Runner.xcodeproj                       # `XCLocalSwiftPackageReference` → umbrella
```

Two structural facts worth pinning down before designing anything:

1. **Plugins are dual-mode by default.** `flutter create` ships both a
   `Package.swift` and a `*.podspec`. The podspec is the CocoaPods fallback
   for consumers that haven't opted into SPM. For our purposes the podspec
   can stay as-is and never be exercised — the example app's
   Runner.xcworkspace consumes SPM only.

2. **`flutter pub get` does NOT bump the umbrella's iOS platform; `flutter
   build` does.** If the plugin requires iOS 26 but the umbrella is still
   pinned at iOS 13, `xcodebuild` fails with:
   ```
   error: The package product 'camera-spike-spm' requires minimum platform
          version 26.0 for the iOS platform, but this target supports 13.0
          (in target 'FlutterGeneratedPluginSwiftPackage' from project
          'FlutterGeneratedPluginSwiftPackage')
   ```
   `flutter build ios --debug --no-codesign` runs the migration that bumps
   the umbrella's `platforms: [.iOS("13.0")]` → `[.iOS("26.0")]`, sourced
   from the Runner.xcodeproj's `IPHONEOS_DEPLOYMENT_TARGET`. Documented in
   `flutter_tools/lib/src/macos/swift_package_manager.dart` — the migration
   only fires if the project deployment target is greater than the default.

3. **Product names are kebab-case, target names snake_case.** The umbrella's
   `.product(name: "camera-spike-spm", package: "camera_spike_spm")` is
   auto-derived from the pubspec name. The plugin's own Package.swift must
   declare `.library(name: "camera-spike-spm", targets: ["camera_spike_spm"])`
   verbatim — change it and Flutter's discovery breaks.

## The Package.swift that works

The plugin's `ios/camera_spike_spm/Package.swift` mirrors the
eva-swift-stitch CameraKit shape verbatim, plus a fourth target for the
Flutter plugin registrar:

```swift
// swift-tools-version: 6.0
import PackageDescription

let package = Package(
    name: "camera_spike_spm",
    platforms: [.iOS("26.0")],
    products: [
        // Kebab-case product name is required — Flutter's auto-generated
        // umbrella references it as `.product(name: "camera-spike-spm", ...)`.
        .library(name: "camera-spike-spm", targets: ["camera_spike_spm"])
    ],
    dependencies: [
        .package(url: "https://github.com/apple/swift-atomics.git", from: "1.2.0"),
    ],
    targets: [
        .target(
            name: "CameraKitCxx",
            dependencies: [],
            publicHeadersPath: "include",
            cxxSettings: [
                .define("CPP_POOL_THREAD_COUNT", to: "4"),
                .headerSearchPath("include"),
            ]
        ),
        .target(
            name: "CameraKitInterop",
            dependencies: [
                "CameraKitCxx",
                .product(name: "Atomics", package: "swift-atomics"),
            ],
            swiftSettings: [
                .swiftLanguageMode(.v6),
                .interoperabilityMode(.Cxx),
            ]
        ),
        .target(
            name: "CameraKit",
            dependencies: [
                "CameraKitInterop",
                .product(name: "Atomics", package: "swift-atomics"),
            ],
            resources: [.process("Shaders")],
            swiftSettings: [
                .swiftLanguageMode(.v6),
                .interoperabilityMode(.Cxx),  // required to consume CameraKitInterop
            ]
        ),
        .target(
            name: "camera_spike_spm",
            dependencies: ["CameraKit"],
            swiftSettings: [
                .interoperabilityMode(.Cxx),
                // Plugin layer stays at Swift 5 to avoid the FlutterMethodNotImplemented
                // Sendable warning that strict Swift 6 concurrency flags. Matches the
                // existing cambrian_camera.podspec choice.
                .swiftLanguageMode(.v5),
            ]
        ),
    ],
    cxxLanguageStandard: .cxx20
)
```

Compile result on `iphoneos` with iOS 26 deployment floor:
```
** BUILD SUCCEEDED **
```

Same `-sdk iphoneos -destination 'generic/platform=iOS' build
CODE_SIGNING_ALLOWED=NO CODE_SIGNING_REQUIRED=NO` command as the CocoaPods
spike. Zero errors, zero warnings beyond template-default boilerplate.

## Build-output receipts

DerivedData (`Build/Products/Debug-iphoneos/`) contains an object per
SPM target and a `.swiftmodule` per Swift target:

```
_AtomicsShims.o                     # apple/swift-atomics C shim
Atomics.o + Atomics.swiftmodule     # apple/swift-atomics Swift module
CameraKitCxx.o                      # C++20 target
CameraKitInterop.o + .swiftmodule   # Swift-Cxx-interop boundary
CameraKit.o + .swiftmodule          # outer Swift module
camera_spike_spm.o + .swiftmodule   # Flutter plugin registrar
FlutterGeneratedPluginSwiftPackage.o + .swiftmodule    # Flutter's umbrella
camera_spike_spm_CameraKit.bundle/   # resource bundle (Metal shader compiled to default.metallib)
Runner.app/                          # final consuming app
```

Note that SPM defaults to **static linkage** here — each target builds as a
`.o` archive, not a `.framework`. Everything ends up linked into
`Runner.app/Runner.debug.dylib` (Flutter's debug dylib pattern).

Symbol receipts confirm the chain links end-to-end:

```bash
$ nm CameraKitCxx.o | grep hello_cpp

0000000000000020 T _hello_cpp_thread_count

$ nm Runner.app/Runner.debug.dylib | grep ManagedAtomic | head -3
00000000000310e4 t _$s7Atomics13ManagedAtomicC4_ptr...      # ManagedAtomic class methods
0000000000030d4c T _$s7Atomics13ManagedAtomicC9__storage...
0000000000030c0c T _$s7Atomics13ManagedAtomicC9__storage...vg
```

Metal shader bundled correctly:
```
Runner.app/camera_spike_spm_CameraKit.bundle/default.metallib
```

Resource bundle naming convention is `<packageName>_<targetName>.bundle`.
Swift code finds it via the standard SPM `Bundle.module` accessor — no
`Bundle(for: BundleAnchor.self).url(...)` gymnastics required.

## SPM vs. CocoaPods — side-by-side cost comparison

| | SPM | CocoaPods |
|---|---|---|
| Three-target shape (Cxx + Interop + outer Swift) | One `Package.swift`, three `.target` blocks | Four sibling `.podspec` files |
| `apple/swift-atomics` | `.package(url: …)` — direct | Vendor sources into two extra sibling pods |
| Resource bundles | `[.process("Shaders")]` + `Bundle.module` | `s.resource_bundles` + `Bundle(for:).url(forResource:in:"…bundle")` at every call site |
| Plugin registration | One product `.library(name: "kebab-case", …)` Flutter auto-discovers | One `s.dependency 'Flutter'` podspec entry |
| Consumer Podfile entries | **Zero** | **Six** `:path =>` entries (or vendored XCFrameworks via Option B) |
| iOS 26 deployment floor | Set once in plugin `Package.swift`; `flutter build` migrates the umbrella | Set in 4+ podspecs + Podfile + Runner.xcodeproj |
| Modulemap authoring | Hand-written `module.modulemap` works (no `framework` keyword needed — SPM doesn't framework-wrap) | Must omit `s.module_map` and let CocoaPods auto-generate `framework module …` |
| Swift 6 / Cxx-interop combo | `.swiftLanguageMode(.v6)` + `.interoperabilityMode(.Cxx)` per target | `'OTHER_SWIFT_FLAGS' => '$(inherited) -cxx-interoperability-mode=default'` per podspec |
| Build system | Native — `xcodebuild` reads `Package.swift` directly | CocoaPods generates `Pods.xcodeproj`; raw `swift build` won't work |

**The SPM path is strictly simpler at every comparison axis.** The CocoaPods
path is open (and verified — see sibling doc), but it pays the cost of
manually re-expressing things SPM gives away for free.

## Caveat 1 — `swift build` standalone (resolved by reasoning, 2026-05-15)

**Status: not a meaningful test for an iOS-only library.**

The eva-swift-stitch CameraKit package already cannot be `swift build`'d.
CLAUDE.md §6 explicitly forbids it:

> `swift build --package-path CameraKit/` and `swift test --package-path …`
> are also forbidden: SPM defaults to the host triple (macOS); CameraKit uses
> iOS-only AVFoundation APIs, the host build fails, and the failure cascades
> into phantom SourceKit "cannot find type Size/WhiteBalanceGains" errors
> across unrelated files.

The migration target inherits this constraint — `swift build` from the
command line wasn't a working path before the migration and isn't one
after. All compilation goes through Xcode (`xcodebuild` directly during
this spike; `XcodeBuildMCP` / `scripts/build-summary.sh` in production).
No need to test what's already known to fail and already not used.

## Caveat 2 — Dynamic linkage (verified 2026-05-15)

**Status: PASS.** Setting the plugin's product to `.dynamic`:

```swift
products: [
    .library(name: "camera-spike-spm", type: .dynamic, targets: ["camera_spike_spm"])
],
```

Rebuild via `flutter build ios --debug --no-codesign` succeeds. The plugin
ships as a dynamic framework inside the final app:

```
Runner.app/Frameworks/camera-spike-spm.framework
```

(vs. the default static build where everything links into `Runner.debug.dylib`
and no `Frameworks/` entry for the plugin exists.)

Flutter's auto-generated umbrella stays at `type: .static` and tolerates a
dynamic plugin product as a transitive dep with no special handling.

**For the migration target, keep `.static` (the default).** It's the
faster cold-start path — no dyld lookups per plugin call. The `.dynamic`
verification matters only as confirmation that the option exists if a
specific reason ever surfaces.

## Caveat 3 — Dual-mode coexistence with a CocoaPods-only plugin (verified 2026-05-15)

**Status: PASS.** Added a synthetic CocoaPods-only sibling plugin
(`legacy_plugin/ios/`: only a `.podspec`, no `Package.swift`) as a
`path:` dep in the example app's `pubspec.yaml`. Flutter cleanly splits
plugins by available build system; the build succeeds end-to-end.

After `flutter pub get` + `flutter build ios --debug --no-codesign`:

**The umbrella does NOT pick up the CocoaPods-only plugin:**

```swift
// FlutterGeneratedPluginSwiftPackage/Package.swift — auto-generated
dependencies: [
    .package(name: "integration_test",   path: "../.packages/integration_test"),
    .package(name: "camera_spike_spm",   path: "../.packages/camera_spike_spm")
    // legacy_plugin absent — no Package.swift to depend on
],
```

**A `Podfile` is auto-generated alongside (not replacing) the umbrella:**

```yaml
# Podfile.lock excerpt
PODS:
  - Flutter (1.0.0)
  - legacy_plugin (0.0.1):
    - Flutter

EXTERNAL SOURCES:
  legacy_plugin:
    :path: ".symlinks/plugins/legacy_plugin/ios"
```

**`GeneratedPluginRegistrant.m` registers SPM and Pods plugins uniformly:**

```objc
+ (void)registerWithRegistry:(NSObject<FlutterPluginRegistry>*)registry {
  [CameraSpikeSpmPlugin registerWithRegistrar:[registry registrarForPlugin:@"CameraSpikeSpmPlugin"]];
  [IntegrationTestPlugin registerWithRegistrar:[registry registrarForPlugin:@"IntegrationTestPlugin"]];
  [LegacyPlugin registerWithRegistrar:[registry registrarForPlugin:@"LegacyPlugin"]];
}
```

**The final app contains both:**

```
Runner.app/camera_spike_spm_CameraKit.bundle/        ← SPM resource bundle
Runner.app/camera_spike_spm_camera_spike_spm.bundle/  ← SPM default bundle
Runner.app/Frameworks/legacy_plugin.framework        ← CocoaPods dynamic framework
```

**Dual-mode "cost" is minimal in practice:**

- Two parallel build pipelines run during `flutter build` (CocoaPods does
  `pod install` + Xcode resolves the SPM graph). Adds ~600 ms in this spike.
- Consumer never edits the `Podfile` manually — Flutter's tooling regenerates
  it on every `pub get`.
- Plugin registration is unified at the `GeneratedPluginRegistrant.m` layer.
- No symbol conflicts: SPM plugins link statically into `Runner.debug.dylib`,
  CocoaPods plugins ship as dynamic frameworks under `Runner.app/Frameworks/`.

**One gotcha worth surfacing.** Flutter derives the registrant's class
name as `<PascalCase(plugin_name)>` plus `Plugin` *only if the last word
isn't already "plugin"*. So `legacy_plugin` → `LegacyPlugin` (NOT
`LegacyPluginPlugin`), while `camera_spike_spm` → `CameraSpikeSpmPlugin`.
Mismatch produces:

```
Semantic Issue (Xcode): Use of undeclared identifier 'LegacyPlugin'
  Runner/GeneratedPluginRegistrant.m:31:3
```

Trivial to fix once you know — but the rule isn't documented in any
template comment.

**Implication for the migration.** If the eva-swift-stitch Flutter app
pulls in any pub.dev plugin that's still CocoaPods-only (most major plugins
shipped Package.swift support during 2024–2025; verify per-plugin against
the actual dep list before migration), the project lands in dual-mode
automatically. This is fine — the build succeeds, the UX cost is one
extra `pod install` per `flutter build`, and no developer-facing
configuration changes.

## Outstanding items (genuinely unexercised)

- **Codesigning + device run.** Spike compiled with `CODE_SIGNING_ALLOWED=NO`;
  installing/running on the connected iPad was not exercised. A run-on-device
  follow-up would close the last unknown.

- **Tooling versions.** Verified on Xcode 26.5 beta, Flutter 3.35.7, swift-tools-version 6.0, SPM enabled via `flutter config --enable-swift-package-manager`. Earlier Flutter versions (3.24–3.27) had experimental SPM that may not pick all this up.

## Recommendation for the migration design

**SPM is the path.** The CocoaPods spike was useful as a fallback but the
SPM spike shows the migration becomes substantially simpler:

- Drop the four-pod CameraKit / Interop / Cxx / plugin shape; replace
  with one `Package.swift` containing three `.target` blocks plus the
  Flutter plugin target.
- Keep `apple/swift-atomics` as an SPM `.package(url:)` dep — same form
  the existing eva-swift-stitch `CameraKit/Package.swift` already uses.
- Keep `Bundle.module` for shader lookup — no migration at call sites.
- Plugin layer stays at Swift 5 mode to avoid the Flutter SDK's
  `FlutterMethodNotImplemented` Sendable hit; CameraKit layers stay at
  Swift 6. Confirmed working under SPM.
- Audit the pub.dev plugin tree the migration target will inherit; if
  any plugin is CocoaPods-only, dual-mode kicks in automatically and works
  without developer intervention (verified — Caveat 3 above). No reason
  to delay SPM migration on this concern.

The CocoaPods path remains documented at
`measurements/cocoapods-cxx-spike/2026-05-15.md` if needed, but with all
three caveats closed there's no remaining technical reason to prefer it
over SPM for this migration.
