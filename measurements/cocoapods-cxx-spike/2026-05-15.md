# CocoaPods × Swift / C++20 / Swift-Cxx interop spike — 2026-05-15

> **Read this first:** a follow-up SPM spike at
> `measurements/flutter-spm-spike/2026-05-15.md` showed Flutter's Swift Package
> Manager integration handles the same three-target shape with substantially
> less overhead (no vendoring of `swift-atomics`, no `Bundle(for:)` migration,
> zero `:path =>` Podfile entries, no manual modulemap authoring). **SPM is the
> recommended migration path.** This document remains correct and useful as a
> fallback if SPM falls over on the actual migration target's plugin tree —
> but don't act on its "PASS, no remaining blockers" verdict in isolation.

---


**Question.** Can a single Flutter iOS plugin's `cambrian_camera.podspec` express
CameraKit's three-target shape:

- **CameraKitCxx** — pure C++20 module with public C-ABI headers
- **CameraKitInterop** — Swift module built with `-cxx-interoperability-mode=default`
  (isolates ADR-13 boundary)
- **CameraKit** — outer Swift module, depends on CameraKitInterop

**Verdict.** **PASS, no remaining blockers.** Expressing Swift /
Swift-Cxx-interop / C++20 as three sibling pods plus a Flutter plugin pod
works: `pod install` accepts it, CocoaPods wires transitive Swift module
visibility correctly, and `xcodebuild build` succeeds end-to-end on iPhoneOS
under both `:linkage => :static` and dynamic `use_frameworks!`.

What was verified:

- Three-target structural shape — **PASS**
- iOS 26 deployment target end-to-end (pods + host app + Flutter machinery) — **PASS**
- `s.resource_bundles` for Metal shaders + `Bundle(for:)` lookup — **PASS**
- `swift-atomics` vendored as two sub-pods (`_AtomicsShims` C shim + `Atomics`
  Swift) with `ManagedAtomic<Bool>` callable from inside a cxx-interop module — **PASS**
- (Escape hatch only) Swift stdlib `Synchronization.Atomic` compiles under
  cxx-interop mode — **PASS** *(documented but not adopted; project stays on `swift-atomics`.)*

Subspecs do **not** produce separate Swift modules; only separate pods do.

### Blockers vs. costs

Every "if X also holds" condition is closed. What remains is migration
**work**, not migration **unknowns**:

| | Status |

## Two fixes that were required during the spike

### Fix 1 — drop the hand-written `module.modulemap`

A plain `module CameraKitCxx { header "HelloCpp.h" export * }` modulemap fails
under `use_frameworks!` (modern Flutter default) with:

```
warning: skipping 'HelloCpp.h' because module declaration of 'CameraKitCxx'
         lacks the 'framework' qualifier
error:   header 'HelloCpp.h' not found
error:   could not build Objective-C module 'CameraKitCxx'
```

Reason: CocoaPods wraps each pod as a `.framework`, so the modulemap must
declare `framework module …`, and the header path resolution happens via the
framework's `Headers/` directory — not via the source-tree path that
`s.module_map` points at.

**Fix:** omit `s.module_map`. CocoaPods auto-generates a correct umbrella-style
declaration:

```c
framework module CameraKitCxx {
  umbrella header "CameraKitCxx-umbrella.h"
  export *
  module * { export * }
}
```

For the SPM Package.swift in `eva-swift-stitch/CameraKit/` we *do* keep the
hand-rolled `module.modulemap` (SPM has different layout assumptions). The
two build systems consume the C++ target differently.

### Fix 2 — let the Flutter plugin stay at Swift 5

Putting the consumer plugin at Swift 6 tripped on `FlutterMethodNotImplemented`
(Flutter framework global, not `Sendable`):

```
CameraSpikePlugin.swift:19:14: error: reference to var
  'FlutterMethodNotImplemented' is not concurrency-safe because it involves
  shared mutable state
```

The existing real `cambrian_camera.podspec` already declares `swift_version = '5.0'`,
so this is the realistic shape: **the Flutter plugin layer stays at Swift 5
even though CameraKit/Interop/Cxx are Swift 6**. Cross-version Swift consumption
works.

The plugin layer still needs `-cxx-interoperability-mode=default`, even at
Swift 5 — CameraKit's emitted `.swiftmodule` references C++-shaped types in
its module interface (synthesized by the cxx-interop frontend pass), so the
consumer must enable the same mode to deserialize the interface.

## CocoaPods wires transitive module visibility correctly

This was the load-bearing risk per the advisor. After `pod install`, the
generated xcconfig files prove CocoaPods walks the dep graph transitively:

`Pods/Target Support Files/camera_spike/camera_spike.debug.xcconfig`:
```ini
FRAMEWORK_SEARCH_PATHS = $(inherited) \
  "${PODS_CONFIGURATION_BUILD_DIR}/CameraKit" \
  "${PODS_CONFIGURATION_BUILD_DIR}/CameraKitCxx" \
  "${PODS_CONFIGURATION_BUILD_DIR}/CameraKitInterop"
OTHER_SWIFT_FLAGS = $(inherited) -D COCOAPODS $(inherited) -cxx-interoperability-mode=default
```

The Flutter plugin pod declares only `s.dependency 'CameraKit'`, yet CocoaPods
adds **all three** sibling frameworks to its search path — full transitive
closure. The cxx-interop flag is also propagated. This is the proof that
multi-pod Swift module visibility works without manual fiddling.

For `use_frameworks!` (both `:linkage => :static` and the default dynamic
linkage) the mechanism is `FRAMEWORK_SEARCH_PATHS` (framework wrapping); for
legacy `use_modular_headers!` it would be `SWIFT_INCLUDE_PATHS`. Both
framework-wrapped linkages were tested end-to-end and produced
`** BUILD SUCCEEDED **`.

### Direct symbol receipts

```bash
$ nm -gU CameraKitCxx.framework/CameraKitCxx | grep hello_cpp
0000000000000000 T _hello_cpp_add
0000000000000020 T _hello_cpp_thread_count

$ nm -u camera_spike.framework/camera_spike | grep CameraKit
_$s9CameraKit5HelloO5greetyS2SFZ      # CameraKit.Hello.greet(_:) undefined → resolved at link via FRAMEWORK_SEARCH_PATHS
```

The C++ symbols are defined in CameraKitCxx; the Flutter plugin's binary
imports `CameraKit.Hello.greet(_:)` (Swift-mangled). The link succeeds.

### Resources cleared the same way

`s.resource_bundles = { 'CameraKitShaders' => ['Sources/*.metal'] }` on the
CameraKit sub-pod produces `CameraKitShaders.bundle` containing a compiled
`default.metallib` (Xcode's Metal compile phase runs on the .metal file).
The bundle is auto-copied into `Runner.app/CameraKitShaders.bundle`. Swift
locates it via:

```swift
private final class BundleAnchor {}
public static var resourceBundle: Bundle {
    let frameworkBundle = Bundle(for: BundleAnchor.self)
    let url = frameworkBundle.url(forResource: "CameraKitShaders",
                                  withExtension: "bundle")!
    return Bundle(url: url)!
}
```

This replaces the SPM `Bundle.module` idiom at every call site that reads a
shader — a mechanical search-and-replace across `MetalPipeline.swift` and
neighbors in the real CameraKit.

## The real cost — Flutter consumer UX

The Flutter plugin's `s.dependency 'CameraKit'` **cannot** point to a
sibling `.podspec` by relative path. CocoaPods resolves named dependencies
through spec repos and Podfile entries; there is no "relative spec path"
syntax inside a podspec.

So the consumer app's `ios/Podfile` must list four `pod ... :path =>` entries:

```ruby
target 'Runner' do
  use_frameworks! :linkage => :static

  pod 'CameraKitCxx',     :path => '<plugin>/ios/CameraKitCxx'
  pod 'CameraKitInterop', :path => '<plugin>/ios/CameraKitInterop'
  pod 'CameraKit',        :path => '<plugin>/ios/CameraKit'

  flutter_install_all_ios_pods File.dirname(File.realpath(__FILE__))
end
```

This breaks the normal Flutter UX where `flutter pub get` auto-wires the
plugin via `pubspec.yaml` — the three inner pods are invisible to Flutter's
auto-discovery. Consumers integrating the plugin must hand-edit their
Podfile, or copy a template snippet from the plugin's `README.md`.

**This is the dominant cost.** Multi-module structure works; the UX hit is
the Podfile boilerplate.

## Addendum — Swift stdlib types compile under cxx-interop (verified 2026-05-15)

**Status: tested escape hatch only.** Project decision: stay on
`swift-atomics`. Evidence below is preserved as proof that stdlib types
co-exist cleanly with `-cxx-interoperability-mode=default` should the
decision ever reverse — not as a recommended migration step.

iOS 18 ships `Synchronization.Atomic<T>` in the stdlib. To prove it doesn't
fight the cxx-interop build mode, the spike added this file to
`CameraKitInterop` (the C++-interop Swift module) and rebuilt:

```swift
import Synchronization

private final class AtomicBoolBox: Sendable {
    let atom = Atomic<Bool>(false)   // ~Copyable, must live inside a reference type
}

public enum AtomicSpike {
    public static func raceOnce() -> Bool {
        let box = AtomicBoolBox()
        let (firstWon, _)  = box.atom.compareExchange(
            expected: false, desired: true, ordering: .sequentiallyConsistent)
        let (secondWon, _) = box.atom.compareExchange(
            expected: false, desired: true, ordering: .sequentiallyConsistent)
        return firstWon && !secondWon
    }
}
```

`** BUILD SUCCEEDED **`. The CAS API surface is essentially identical to
`ManagedAtomic` (`compareExchange(expected:desired:ordering:)` →
`(exchanged, original)`); the difference is that `Atomic<T>` is `~Copyable`
and must live in a reference holder if shared across closures.

**Generalisable result:** Swift stdlib types (including `~Copyable` and
`Sendable`-constrained ones) participate normally in a cxx-interop module
under CocoaPods. No special handling required.

## Addendum — `swift-atomics` vendored as two sub-pods (verified 2026-05-15)

### Why vendor — there is no published podspec to install

`apple/swift-atomics` is **SPM-only**: the upstream repo ships `Package.swift`
+ `CMakeLists.txt`, no `.podspec` at any tag. That eliminates two paths a
maintainer would normally reach for:

- `pod trunk info Atomics` returns `biasedbit/Atomics 0.9.0 (2015)` — an
  unrelated OSAtomic-wrapper package. `pod trunk info SwiftAtomics` and
  `swift-atomics` return "No pod found." There is no community-maintained
  podspec for Apple's package on CocoaPods trunk.
- `pod 'Atomics', :git => 'https://github.com/apple/swift-atomics.git',
  :tag => '1.2.0'` would fail at install time — CocoaPods requires a
  `.podspec` at the git ref, which `apple/swift-atomics` does not provide.

That leaves three theoretical alternatives, none of which beat vendoring for
this use case:

| Alternative | Why it loses |
|---|---|
| Publish our own `Atomics.podspec` to CocoaPods trunk | Community-impacting; we'd own version skew with upstream forever |
| Maintain a private CocoaPods spec repo | Heavier infra than vendoring; consumers need `source 'git@…/specs.git'` in their Podfile |
| Wait for an official Apple podspec | Not on the swift-atomics roadmap; SPM is the supported path |

**Vendor the sources** at the pinned upstream tag (1.2.0, Apache 2.0). The
package is small (~20 .swift files + 1 .c shim, no transitive deps), and
this aligns with how other Apple-SPM libraries (`swift-collections`,
`swift-log`) typically reach CocoaPods consumers.

### What we vendored

Two sibling pods that mirror the SPM target structure exactly:

```
ios/
├── AtomicsShims/                    # `s.module_name = '_AtomicsShims'`
│   ├── AtomicsShims.podspec         # C target — _AtomicsShims.c + .h
│   └── Sources/
│       ├── src/_AtomicsShims.c
│       └── include/_AtomicsShims.h
└── Atomics/
    ├── Atomics.podspec              # Swift target — depends on AtomicsShims
    └── Sources/
        ├── Types/{ManagedAtomic, UnsafeAtomic, DoubleWord, ...}.swift
        ├── Protocols/{AtomicInteger, AtomicValue, ...}.swift
        ├── Conformances/{RawRepresentable, OptionalRawRepresentable, ...}.swift
        ├── Primitives/autogenerated/{Primitives.native, Primitives.shims}.swift
        └── …                         # 20 .swift files total; .gyb templates excluded
```

Two non-obvious bits that the spike had to discover:

1. **`s.module_name = '_AtomicsShims'`** on the C-shim pod. Pod names can't
   start with an underscore, but `import _AtomicsShims` from Atomics's Swift
   sources is hard-coded. `s.module_name` decouples pod name from clang
   module name.

2. **Mirror the upstream Swift / C build settings**:
   ```ruby
   # Atomics.podspec
   'OTHER_SWIFT_FLAGS' => '$(inherited) -D ATOMICS_NATIVE_BUILTINS \
                                       -enable-experimental-feature BuiltinModule'

   # AtomicsShims.podspec
   'GCC_PREPROCESSOR_DEFINITIONS' => '$(inherited) ATOMICS_NATIVE_BUILTINS=1'
   ```
   Without these the build halts at:
   ```
   error: swift-atomics requires native builtins on Swift 5.9
   ```
   Source-guard in `Primitives/autogenerated/Primitives.native.swift:26` and
   `Types/DoubleWord.swift:18`. These flags are set on the upstream SPM
   target via `_swiftSettings` / `_cSettings` in `swift-atomics/Package.swift`.

After applying both fixes: `** BUILD SUCCEEDED **`. The `AtomicSpike.swift`
test file in `CameraKitInterop` mirrors `AsyncWithTimeout.swift:21–28`:

```swift
import Atomics

public enum AtomicSpike {
    public static func raceOnce() -> Bool {
        let resumed = ManagedAtomic<Bool>(false)
        let (firstWon,  _) = resumed.compareExchange(
            expected: false, desired: true, ordering: .sequentiallyConsistent)
        let (secondWon, _) = resumed.compareExchange(
            expected: false, desired: true, ordering: .sequentiallyConsistent)
        return firstWon && !secondWon
    }
}
```

Symbol receipts confirm the chain links end-to-end:

```bash
$ nm -gU AtomicsShims/_AtomicsShims.framework/_AtomicsShims
T __sa_retain_n        # upstream C retain shim
T __sa_release_n       # upstream C release shim

$ nm -u CameraKitInterop/CameraKitInterop.framework/CameraKitInterop | grep Atomics
_$s7Atomics13ManagedAtomicCMa     # ManagedAtomic class metadata access
_$s7Atomics13ManagedAtomicCMn     # ManagedAtomic nominal type descriptor
```

**Migration impact on real CameraKit.** No source change at `ManagedAtomic`
call sites — the API surface is identical between SPM `apple/swift-atomics`
and the vendored sources. The migration cost is purely build-system:

- Drop `dependencies: [.package(url: "https://github.com/apple/swift-atomics.git", from: "1.2.0")]`
  from `Package.swift`.
- Add `AtomicsShims/` and `Atomics/` sub-pods under the Flutter plugin's
  `ios/` directory; vendor the upstream sources at the same 1.2.0 tag.
- Add `pod 'AtomicsShims', :path => …` and `pod 'Atomics', :path => …` to
  the consumer Podfile (now 6 `:path =>` entries total, up from 4).
- Document upstream-version pin in the plugin's `README.md`; refresh by
  re-copying sources when bumping.

## Workarounds, ranked

If the four-Podfile-entry UX is unacceptable, options in order of effort:

### Option A — collapse to **two pods** (Flutter plugin + one CameraKit)
Merge `CameraKitCxx` + `CameraKitInterop` + `CameraKit` into a single pod.
The ADR-13 cxx-interop containment is preserved by *file-level* organization
(only `Sources/Interop/*.swift` files declare `-cxx-interoperability-mode=default`
via a per-file flag, or — easier — the whole module turns on cxx-interop but
public C++ types are not re-exported from `CameraKit.h`).

Loses: enforced compile-time isolation of the interop seam.

### Option B — ship pre-built **XCFrameworks**
`CameraKitCxx.xcframework`, `CameraKitInterop.xcframework`, `CameraKit.xcframework`
are pre-built and dropped under `ios/Frameworks/`. The plugin's podspec uses
`s.vendored_frameworks = 'Frameworks/*.xcframework'`. One Podfile entry only.

Loses: source-level builds for plugin contributors; release pipeline must
emit XCFrameworks (script wraps `xcodebuild archive` per arch + `xcodebuild
-create-xcframework`).

Gains: preserves three modules; one Flutter-plugin-shaped artifact.

### Option C — accept the four `:path =>` entries
Document them in `README.md` and `CHANGELOG.md`. Provide a `pod-prelude.rb`
helper that the consumer's Podfile `eval_file`s. Lowest engineering cost,
worst initial-setup friction.

**Recommendation.** If the migration target ships as a public plugin: **B**
(XCFrameworks). If internal-only with controlled consumers: **C** (document
the four entries). **A** is only attractive if XCFramework distribution
proves too heavy.
