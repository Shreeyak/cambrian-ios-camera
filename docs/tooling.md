# CameraKit Tooling

Small tools that reduce coordinator/subagent context burn and enforce the
stage-discipline conventions. Each tool has one job. Reach for the right
tool from the decision tree below; don't improvise.

## Decision tree

```
Need to …                                 │ Use
──────────────────────────────────────────┼───────────────────────────────
Understand a file's public API            │ LSP documentSymbol (MCP),
                                          │   or scripts/lsp-symbol.sh
Look up a signature by name               │ LSP workspaceSymbol (MCP)
Find references / who-calls-X             │ LSP prepareCallHierarchy (MCP)
Find all scaffold slugs                   │ scripts/scaffold-inventory.sh
Search for a literal string / pattern     │ Grep tool (Claude) or `rg`
Build & verify iOS target                 │ scripts/build-summary.sh
Orient yourself at stage kickoff          │ CameraKit/CONTRACTS.md
                                          │   + scripts/stage-preflight.sh
Refresh CONTRACTS.md after source changes │ scripts/regen-contracts.sh
                                          │   (auto via pre-commit hook)
Stream live CONTRACTS regen while editing │ scripts/watch-contracts.sh
```

---

## Canonical docs (always read first)

### `CameraKit/CONTRACTS.md` — "what exists right now"

Auto-generated snapshot of the CameraKit library: active scaffold slugs +
compressed public-API signatures for every source file. **This is every
subagent's first read.** It replaces what used to be 500-line briefs
inlined into agent prompts.

Regenerated by:
1. `scripts/regen-contracts.sh` — explicit full regen
2. `scripts/watch-contracts.sh` — live loop during editing
3. `.git/hooks/pre-commit` — staged automatically before each commit

Never edit by hand; changes are overwritten.

### `CameraKit/DECISIONS.md` — stigmergy log

Append-only. Agents write one-line entries for decisions or assumptions
that would otherwise clutter their return text:

```
2026-04-21 [stage-02 task-03] impl-agent — chose sessionQueue.async over .sync per ADR-30
```

The coordinator doesn't re-read this during a stage. The next subagent
glances at it before its task. At stage boundaries, fold entries into
`state.md`'s "Decisions taken that weren't in briefs" section and
truncate below the stage separator.

## Scripts

All scripts live under `scripts/`, are ≤80 lines, and exit non-zero on
failure. They assume the repo root as CWD.

### `scripts/build-summary.sh`

Wraps `xcodebuild` and returns a 3-line summary instead of SwiftDriver
flag dumps.

```bash
$ scripts/build-summary.sh
BUILD: success
Swift errors: 0
Metal errors: 0
Warnings: 1
```

- Auto-detects a scheme-valid iPad simulator via `xcodebuild -showdestinations`.
- `--verbose` passes full xcodebuild output through (escape hatch).
- `--scheme NAME` overrides the default scheme.
- `--destination DEST` pins an explicit destination (device UDID, sim id).

Use this instead of raw `xcodebuild` whenever you only need pass/fail.

### `scripts/scaffold-inventory.sh`

Emits active scaffold slugs as a markdown table.

```bash
$ scripts/scaffold-inventory.sh
| Slug | File:line |
|------|-----------|
| `01:naive-scenephase-stop` | `CameraKit/Sources/CameraKit/CameraEngine.swift:153` |
| `01:simple-metal-passthrough` | `CameraKit/Sources/CameraKit/MetalPipeline.swift:21` |
| `01:simple-metal-passthrough` | `CameraKit/Sources/CameraKit/TexturePoolManager.swift:36` |
| `01:skip-completion-guard` | `CameraKit/Sources/CameraKit/MetalPipeline.swift:115` |
```

Consumed by `regen-contracts.sh` and `stage-preflight.sh`. Runs against
`CameraKit/Sources/` by default; pass a path to target a subtree.

### `scripts/regen-contracts.sh`

Full regeneration of `CameraKit/CONTRACTS.md`.

```bash
$ scripts/regen-contracts.sh
wrote /Users/shrek/.../CameraKit/CONTRACTS.md (552 lines)
```

Strategy: `repomix --compress --remove-comments --no-file-summary
--no-directory-structure` over `CameraKit/Sources/CameraKit/**/*.swift`,
prepended with the scaffold inventory.

**v2 candidate:** compiler-emitted `.swiftinterface` files (via
`-emit-module-interface` in Package.swift `swiftSettings`). Verbose but
100% correct. Deferred until v1 proves insufficient.

### `scripts/regen-contracts-partial.sh <path>`

Called by `scripts/watch-contracts.sh` on file save. v1 defers to full
regen (repomix is fast enough). Exists so we can drop in a smarter
splice-in-place implementation later without changing the watcher.

### `scripts/stage-preflight.sh`

Validates the repo is in a coherent state before a stage kicks off.
Checks:

1. Every slug in `state.md` under "Scaffolding still live" has ≥1 hit
   in source.
2. Every slug in source is documented in `state.md`.
3. `CONTRACTS.md` exists and is ≤24h old (regenerates if not).
4. `build-summary.sh` succeeds.

Prints `=== PASS ===` or `=== FAIL ===`. Exit 0 / 1.

### `scripts/watch-contracts.sh`

Long-running. Uses `fswatch` to regenerate CONTRACTS.md on every save
under `CameraKit/Sources/`. Run in a tmux pane while editing:

```bash
$ scripts/watch-contracts.sh
watching /Users/shrek/.../CameraKit/Sources — CONTRACTS.md will refresh on save
```

### `scripts/lsp-symbol.sh`

One-shot sourcekit-lsp client for scripted queries. Prefer the `LSP`
MCP tool for in-session use; reach for this shell wrapper for batch
or when the MCP tool isn't available.

```bash
# File outline
scripts/lsp-symbol.sh outline CameraKit/Sources/CameraKit/Constants.swift

# Hover at a position (1-based line/col, editor coordinates)
scripts/lsp-symbol.sh hover CameraKit/Sources/CameraKit/CameraEngine.swift 34 12

# Project-wide symbol search
scripts/lsp-symbol.sh workspace CameraEngine
```

**Known limitation:** standalone sourcekit-lsp crashes on actor-heavy
files when it can't resolve SwiftPM's `Bundle.module` or AVFoundation
imports (`Illegal instruction: 4` on CameraEngine.swift). It works fine
on leaf files (Constants.swift, value types). For full-project LSP use
the MCP tool — it auto-configures via BSP through `buildServer.json`.

## MCP tools (preferred for in-session use)

### `LSP` — sourcekit-lsp via the MCP layer

Semantic code queries without the standalone wrapper's limitations.
Takes `operation` + `filePath` + `line`/`character` (1-based). Wires
into the `buildServer.json` BSP bridge automatically.

Useful operations for context-efficient exploration:

- `documentSymbol` — file outline in ~15 lines vs. 200 lines of source.
- `hover` — signature + docs for a symbol.
- `workspaceSymbol` — find a declaration project-wide.
- `prepareCallHierarchy` + `incomingCalls` — who calls X (cross-file).
- `goToDefinition` / `goToImplementation` — jump to decl or impl site.

### `XcodeBuildMCP` — build / run / test / simulator control

Session defaults are already set (project, scheme, device). Call
`session_show_defaults` first each session, then prefer the high-level
tools (`build_run_device`, `test_sim`, etc.) over raw `xcodebuild`.

### `xcode` — the running-Xcode integration

Only for actions that require Xcode itself (navigator issues, preview
rendering, the open window). Reach for it rarely. `XcodeListNavigatorIssues`
is the one action with no alternative.

## One-time host setup

```bash
brew install xcode-build-server fswatch swift-format ripgrep repomix
xcode-build-server config -project eva-swift-stitch.xcodeproj \
                          -scheme eva-swift-stitch
```

The `xcode-build-server config` step creates `buildServer.json` at the
repo root, bridging sourcekit-lsp to Xcode's build system. That file is
gitignored (contains a machine-specific DerivedData path). Re-run
`xcode-build-server config ...` after cloning or switching schemes.

## Conventions

- Tools are ≤80 lines. If a script grows, split it — don't let it become
  a framework.
- Every tool emits plain text or markdown. No interactive prompts, no
  TUI, no colors beyond what xcodebuild naturally produces.
- Exit code is the contract: 0 success, non-zero failure.
- Scripts assume repo root as CWD. They use `$(cd "$(dirname "$0")/.."
  && pwd)` for paths that need absolute resolution.
- Stage-specific behavior lives in `CameraKit/state.md`, not in the
  scripts. Scripts are stage-agnostic.

## Changing a script

Before editing:
1. Read the plan at `.claude/plans/jolly-pondering-tiger.md` — the
   script design constraints are documented there.
2. Regenerate and check `CONTRACTS.md` after the change to verify no
   regression in coordinator workflow.
3. If the change adds a new dependency, update this doc and the CLAUDE.md
   one-time-setup block.
