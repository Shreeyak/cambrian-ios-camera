# CameraKit Tooling

Small tools that reduce coordinator/subagent context burn. Each tool has one
job. Reach for the right one from the decision tree below; don't improvise.

> Quick index of every script lives in `scripts/CLAUDE.md` (auto-loaded when
> working under `scripts/`). This doc is the deeper "what / when / why".

## Decision tree

```
Need to …                                  │ Use
───────────────────────────────────────────┼──────────────────────────────────
See "what exists right now" (current API)  │ CameraKit/CONTRACTS.md
Understand a file's symbols / outline       │ LSP documentSymbol (MCP),
                                           │   or scripts/lsp-symbol.sh outline
Look up a signature / find a declaration    │ LSP workspaceSymbol (MCP)
Find references / who-calls-X               │ LSP prepareCallHierarchy (MCP)
Check isolation / Sendable / @available     │ scripts/dump-interface.sh
Search for a literal string / pattern       │ Grep tool (Claude) or `rg`
Refresh CONTRACTS.md after source changes   │ scripts/regen-contracts.sh
                                           │   (auto via the pre-commit hook)
Build & verify the iOS target               │ XcodeBuildMCP build_run_device,
                                           │   or scripts/build-summary.sh
Build the app & launch it on the iPad       │ scripts/build-launch.sh
Run the tests                               │ XcodeBuildMCP test_device,
                                           │   or scripts/test-summary.sh
Wire a new test file into the Xcode target  │ scripts/sync-test-target.sh
Read app logs off the iPad                  │ ipad-logs skill
                                           │   (scripts/device-log-live.sh)
```

**Device-only on this machine.** No simulators (memory constraint — see
CLAUDE.md §6). Every build/run/test/log tool targets a physical iPad first,
then Mac "Designed for iPad", then errors. The device-targeting scripts
(`build-launch.sh`, `device-log-live.sh`) auto-detect the connected paired iPad.

## Canonical docs (always read first)

### `CameraKit/CONTRACTS.md` — "what exists right now"

Auto-generated snapshot of the CameraKit library: compressed public-API
signatures for every source file. **This is every subagent's first read.**

Regenerated by:
1. `scripts/regen-contracts.sh` — explicit full regen
2. The lefthook **pre-commit** hook — staged automatically before each commit

Never edit by hand; changes are overwritten. It's byte-deterministic (no
embedded timestamp), so the hook only produces a diff on a real API-shape
change.

### `CameraKit/DECISIONS.md` — stigmergy log

Append-only. Agents write one-line entries for decisions or assumptions that
would otherwise clutter their return text:

```
2026-04-21 [task-03] impl-agent — chose sessionQueue.async over .sync per ADR-30
```

The coordinator doesn't re-read this mid-task; the next subagent glances at it
before its task.

## Scripts

All scripts live under `scripts/`, are small and single-purpose, assume the
repo root as CWD, and exit non-zero on failure.

### `scripts/build-launch.sh`

Build the app and install + launch it on an auto-detected paired iPad.

```bash
scripts/build-launch.sh            # Debug   (default)
scripts/build-launch.sh --release  # Release
```

Builds with `xcodebuild` (generic iOS destination, full log under
`.build-logs/`), then installs + launches via `devicectl`. The device is
auto-discovered — the project's two iPads rotate one at a time, so the first
reachable paired iPad wins. iPad-only; errors if none is reachable.

### `scripts/build-summary.sh`

Wraps `xcodebuild build` and returns a concise pass/fail summary instead of
SwiftDriver flag dumps. Fallback for when XcodeBuildMCP is unavailable.

```bash
$ scripts/build-summary.sh
DEST: physical iPad <udid>
BUILD: success
Errors: 0  Warnings: 0
Log:  .build-logs/<ts>-build-eva-swift-stitch.log
JSON: .build-logs/<ts>-build-eva-swift-stitch.json
```

- Resolves the destination: physical iPad → Mac "Designed for iPad" → error
  (never a simulator).
- Pipes `xcodebuild` through `xcsift`; persists raw log + structured JSON under
  `.build-logs/`. Read the JSON first on failure — it has file/line/message.
- `--scheme NAME`, `--destination DEST`, `--verbose` (full log passthrough).

### `scripts/test-summary.sh`

Wraps `xcodebuild test`. Same logging/destination behavior as `build-summary`.

```bash
$ scripts/test-summary.sh --filter eva-swift-stitchTests/Stage12CoordinatorTests
TESTS: success
Build errors: 0  Warnings: 0  Failed cases: 0
```

- Defaults to scheme `eva-swift-stitch` (the app-hosted target that runs
  CameraKitTests on device via dual-membership — see CLAUDE.md §8).
- `--filter SUITE` → `-only-testing:<SUITE>`. Each `@Suite` is its own struct,
  so filter by the struct name, not the filename.

### `scripts/regen-contracts.sh`

Full regeneration of `CameraKit/CONTRACTS.md`.

```bash
$ scripts/regen-contracts.sh
wrote /Users/.../CameraKit/CONTRACTS.md (2129 lines)
```

Strategy: `repomix --compress --remove-comments --no-file-summary
--no-directory-structure` over `CameraKit/Sources/CameraKit/**/*.swift`,
then sorts the per-file blocks for deterministic output. Auto-runs on the
pre-commit hook; rarely needed by hand.

**v2 candidate:** compiler-emitted `.swiftinterface` files (via
`-emit-module-interface`). Verbose but 100% correct. Deferred until v1 proves
insufficient.

### `scripts/sync-test-target.sh`

Dual-membership wiring for CameraKitTests. Re-runs the xcodeproj-side wiring
that compiles every `.swift` file in `CameraKit/Tests/CameraKitTests/` inside
the app-hosted `eva-swift-stitchTests` Xcode target (the one that runs on the
iPad). The SwiftPM `testTarget` in `Package.swift` is left untouched. Idempotent.

```bash
touch CameraKit/Tests/CameraKitTests/NewThingTests.swift
scripts/sync-test-target.sh        # add it to the Xcode target, then run tests
```

Drives `project.pbxproj` through the `xcodeproj` gem — never hand-edit the
pbxproj. See CLAUDE.md §8 for the why.

### `scripts/dump-interface.sh`

Emits `/tmp/CameraKit.swiftinterface` (or `$1`) — the compiler-validated public
API surface of the CameraKit module.

```bash
$ scripts/dump-interface.sh
wrote /tmp/CameraKit.swiftinterface (465 lines, public API only, compiler-validated)
```

Complements `CONTRACTS.md` — see CLAUDE.md §6.2 for "when to use each". In
short: `.swiftinterface` is authoritative for isolation semantics
(`@MainActor`, `nonisolated`), Sendable conformances, compiler-synthesized
members, and exact `@available` annotations. `CONTRACTS.md` is better for
internal wiring.

Known quirk: `Bundle.module` references produce a compile error at invocation,
but the interface is emitted before the error fires — the script swallows it
and reports success iff the interface file was produced.

### `scripts/lsp-symbol.sh`

One-shot sourcekit-lsp client for scripted queries. Prefer the `LSP` MCP tool
for in-session use; reach for this wrapper for batch use or when the MCP tool
isn't available.

```bash
scripts/lsp-symbol.sh outline   CameraKit/Sources/CameraKit/Constants.swift
scripts/lsp-symbol.sh hover     CameraKit/Sources/CameraKit/CameraEngine.swift 34 12
scripts/lsp-symbol.sh workspace CameraEngine
```

**Known limitation:** standalone sourcekit-lsp crashes on actor-heavy files
when it can't resolve `Bundle.module` or AVFoundation imports (`Illegal
instruction: 4` on `CameraEngine.swift`). Reliable on leaf files only
(`Constants.swift`, value types). For full-project LSP use the MCP tool.

### `scripts/device-log-live.sh`

Backend for the `ipad-logs` skill — the only working device-log path since
iOS 26.4 broke the classic routes. Polls `<Documents>/camerakit.log` off an
auto-detected iPad every 4s via `devicectl device copy from` and mirrors new
lines to `${TMPDIR}/camerakit-live.log`.

```bash
scripts/device-log-live.sh         # start background polling (auto-detects iPad)
scripts/device-log-live.sh tail    # follow the local mirror
scripts/device-log-live.sh grep EXPR
scripts/device-log-live.sh stop
```

Requires the app to call `CameraKitLog.enableFileLogging()` at startup (it
does). Whenever the user asks for device logs, invoke the `ipad-logs` skill.

## MCP tools (preferred for in-session use)

### `LSP` — sourcekit-lsp via the MCP layer

Semantic code queries without the standalone wrapper's limitations. Takes
`operation` + `filePath` + `line`/`character` (1-based). Wires into the
`buildServer.json` BSP bridge automatically.

- `documentSymbol` — file outline in ~15 lines vs. 200 lines of source.
- `hover` — signature + docs for a symbol.
- `workspaceSymbol` — find a declaration project-wide.
- `prepareCallHierarchy` + `incomingCalls` — who calls X (cross-file).
- `goToDefinition` / `goToImplementation` — jump to decl or impl site.

### `XcodeBuildMCP` — build / run / test / device control

Call `session_show_defaults` first each session, then prefer the high-level
**device** tools (`build_run_device`, `test_device`) over raw `xcodebuild`.
**Never** the `*_sim` variants — simulators are disallowed on this machine.

### `xcode` — the running-Xcode integration

Only for actions that require Xcode itself (navigator issues, preview
rendering, the open window). Reach for it rarely.

## One-time host setup

```bash
brew install xcode-build-server swift-format ripgrep repomix xcsift jq lefthook
xcode-build-server config -project eva-swift-stitch.xcodeproj \
                          -scheme eva-swift-stitch
lefthook install
```

- `xcode-build-server config` creates `buildServer.json` at the repo root,
  bridging sourcekit-lsp to Xcode's build system. Gitignored (machine-specific
  DerivedData path); re-run after cloning or switching schemes.
- `lefthook install` installs the pre-commit (swift-format + CONTRACTS regen)
  and pre-push (camerakit-only sync) hooks into `.git/hooks/`. See CLAUDE.md
  §6.0 / §10. Do **not** set `core.hooksPath`.

## Conventions

- Each tool is small and single-purpose. If a script grows into a framework,
  split it.
- Every tool emits plain text or markdown — no interactive prompts, no TUI.
- Exit code is the contract: 0 success, non-zero failure.
- Scripts assume repo root as CWD; use `$(cd "$(dirname "$0")/.." && pwd)` for
  paths needing absolute resolution.

## Changing a script

1. Regenerate and check `CONTRACTS.md` after the change to verify no regression
   in the coordinator workflow.
2. If the change adds a new dependency, update this doc, `scripts/CLAUDE.md`,
   and the CLAUDE.md §6.0 one-time-setup block.
