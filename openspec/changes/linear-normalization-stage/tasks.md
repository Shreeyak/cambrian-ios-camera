## 7. Kernel fusion (profile-driven)

- [x] 7.1 Profiled on device (Shreeyak's iPad, Release, 2026-06-26). Total GPU wall-time of the single shared command buffer (`gpuEndTime − gpuStartTime`, whole pointwise chain + tracker + NV12 — measured through the GPU's own timestamps, no command-buffer splits that would rig the baseline; flag-gated `gpuProfilingEnabled` profiler in `MetalPipeline`'s completion handler). **At full-frame 4032×3024 (heaviest; demo default), no recording: ~12 ms avg, ~17–18 ms max settled** (one 22.6 ms launch-time outlier) vs the **33.3 ms** budget — **~21 ms (~63%) headroom**. NV12 (recording) + tracker would add only ~2–3 ms, so headroom holds. Conclusion: the pipeline is **comfortably under budget even at 12 MP**, so fusion is a power/thermal-headroom optimization, not a correctness/budget necessity (see §7.2 go/no-go).
- [~] 7.2 **Deferred** (user decision 2026-06-26, on the §7.1 data). The fusion shape is mapped — one kernel reading YUV once and writing all three live intermediates (`naturalTex` calibration tap + `processedTex16F` for NV12/still + BGRA8 for tracker/delivery), eliminating the two full-frame RGBA16F re-reads (grade re-reading natural, pack re-reading processed); tracker (MPS) and NV12 stay separate; a no-pack PSO variant covers the `packedTex == nil` still path. Estimated saving ~1–3 ms of GPU load. **Not done because §7.1 shows ~21 ms headroom even at 12 MP** — fusion is a power/thermal optimization, not a budget necessity. Revisit if a heavier pipeline (higher res/fps, more lanes) or a sustained-thermal need appears. **Re-profiled 2026-07-07 at the 4K60 case enabled this session (3840×2160, 8.3 MP) vs the halved 16.6 ms/frame 60-fps budget — settled avg ~9.8 ms, typical max ~11 ms (~34% headroom), rare spikes ~15.8 ms; fits comfortably, so the deferral holds even at 4K60.** (4K60 is fewer pixels than 7.1's 4032×3024, so per-frame GPU time scales down.) **Decisive: the pipeline is crop-first** — the decode kernel (`yuvToRgba`) reads only the `cropOrigin`-offset sub-region and the output textures ARE the crop size, so grade/pack/tracker/NV12 never touch discarded pixels. Profiled the **production config (1024×1024 center crop @ 4K60 + tracker): settled avg ~1.7 ms, max ~4 ms — ~90% headroom** (~1 MP vs the uncropped 8.3 MP). The fusion's ~1–3 ms saving is noise here; it would only ever matter for a sustained UNCROPPED full-frame + recording pipeline, which production doesn't use. Re-open only if such a config appears.
- [x] 7.2b **Fusion A implemented 2026-07-07** (user decision to land it despite the crop-first "~noise" saving above — it's the clean canonical shape and removes the two RGBA16F re-reads). `yuvGradedFused` (ColorShaders.metal) does decode→normalize→grade→pack in registers from a single YUV read; the pointwise grade math is extracted into a shared `gradePixel` helper called by BOTH the fused kernel and the retained `colorTransform` (so the grade golden tests transitively cover the fused grade). Two PSOs via a `kWritePacked` function constant — `yuvGradedFusedPackPSO` (tracker/mailbox/still) and `yuvGradedFusedNoPackPSO` (`packedTex == nil` streaming). `encodeGradedCore` rewired to one encoder; `renderFrame` + `renderStill` both use it unchanged. `decodeYuvBt601`/`CropUniform` are duplicated into ColorShaders.metal (Metal has no cross-file shared header on the SwiftPM build) with KEEP-IN-SYNC notes. **NOT byte-identical to the old path by construction** — fused grades the f32 decode register; the old path graded the 16F texture read-back — so processed differs by ≲1 16F ULP (~5e-4); `naturalTex` IS byte-identical (both store 16F(decode)). Scope A only: processed16F still written unconditionally (skipping it is B) and naturalTex written every frame (on-demand is C) — both remain future micro-opts.
- [x] 7.3 **Acceptance gate met.** Existing grade/normalization golden tests (`colorPipelineGoldenFrame`, `fusedNormalizationPatchCases`, `normalizationAffineInLinearLight`, off-path identity) pass unchanged (they drive `gradePixel` via `colorTransform`), PLUS two new equivalence tests (Stage04Tests) that run the OLD three-encoder core and the fused core on the SAME y/cbcr: `fusedVsSeparateCoreEquivalence` (chroma-rich + full grade) asserts fused natural ≡ separate natural (1e-4) ≡ hand-computed BT.601 (guards the duplicated decode's COEFFICIENTS) and fused processed ≡ separate processed (1e-3); `fusedCropOriginMatchesSeparate` (cropped, non-square origin, luma gradient) guards the duplicated decode's CROP-ORIGIN mapping. 13/13 Stage04 pass on device (Shreeyak's iPad, 2026-07-07). **Remaining HITL:** on-device still-capture visual (the fused Pack variant is exercised by the equivalence test, but the still's returned BGRA8 is a trivial clamp of the asserted-equivalent processed, so this is a visual sanity check, not a correctness gap).
- [x] 7.4 **Measured the realized saving (in-session A/B, 2026-07-07, Shreeyak's iPad, Debug).** `fusedCoreThroughputBenchmark` (Stage04Tests) times the OLD three-encoder core vs the fused core back-to-back at the production **1024² crop** (same thermal state, 200 iters, chroma-rich input + full grade): **separate ≈ 938 µs/frame, fused ≈ 366 µs/frame → ~571 µs saved, a ~61% reduction of the pointwise CORE's GPU time.** This is the *core in isolation* (decode+grade+pack); the full frame at 1024² also runs tracker (MPS) + NV12, measured ~1.7 ms in §7.2 — so the core saving is ~34% of total per-frame GPU. **Correction to §7.2's "~noise" wording:** the saving is NOT noise — it's a real, repeatable ~61% cut of the core, dominated by removing the two full-frame RGBA16F re-reads (memory-bandwidth-bound, so it holds in Release too). §7.2's *conclusion* still stands (the pipeline had ~90% headroom, so fusion was never budget-*required*) — but it IS a meaningful power/thermal win for sustained recording, which is why landing A was worthwhile.
- [x] 7.5 **B (retire 16F processed) + C (natural tap on-demand) implemented + measured (2026-07-07).** B unifies every graded consumer (preview, delivery, NV12 recorder, diagnostic) on the BGRA8 `packed` surface and deletes the per-frame 16F processed write + the `sampleCenterPatch` diagnostic; C gates the 16F natural (calibration) tap behind a `kWriteNatural` function constant, armed only during `calibrateWhite`/`calibrateBlack`. Three-way `fusedCoreThroughputBenchmark` at 1024² (same run; GPU wall-time = config-independent, so these are the Release GPU numbers): **separate 1233 µs/frame → fusedArmed (A+B) 331 µs (−73.2%) → fusedSteady (A+B+C, the 20-min hot path) 273 µs (−77.8%)**; C alone is −17.4% of the armed core. So the steady-state pointwise core is **~4.5× cheaper** than pre-optimization, dominated by eliminating full-frame RGBA16F writes/re-reads (bandwidth-bound). Verified: 24 pipeline + 19 calibration logic tests + device still-path green; commits A `4f11683`, B `fe68ed7`, C `99f3933`. **HITL remaining:** end-to-end record→play (NV12-from-BGRA8) and calibrate on a real white/dark field; a true GPU-utilization/power figure via Instruments Metal System Trace (the per-frame µs above is the load proxy). **Specific C watch-item:** `awaitNaturalRefresh` (arm→PTS-advance→sample) is exercised only as two disconnected halves — the gating test proves an armed frame writes the tap but not that `latestNaturalPTSNs` advanced, and the calibration logic tests sample an *installed* texture without arming. So the arm→await→sample seam runs first on a real calibrate. Failure mode is **graceful** (monotonic camera PTS > 0 returns in ~1 frame; if it somehow didn't advance, `awaitNaturalRefresh` burns its 1s timeout then samples the latest — a delay, never a crash or stale-corrupted calibration).
